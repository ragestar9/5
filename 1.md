● Reviewed the whole project: index.html (1,196 lines), src/main.js (1,884
  lines), src/index.css, src/lib/motion.js, plus configs. I also verified every
  $("#id") referenced in JS exists in the HTML (all present), that the
  [data-module] DOM order matches the MODULES array (it does — the index-based
  nextMod/portstack logic is safe), and that animejs@4.5.0 really exports
  spring/stagger/animate (it does).

  Overall: this is a well-built piece of work. The architecture is sound — one
  telemetry store, one master rAF with an updaters registry, visibility-gated
  renderers, per-frame guards via getBoundingClientRect early returns,
  transform-only writes. The previously-marked FIXes (#1–#6) all look correctly
  implemented. Below are the real issues I found, by severity.

  Bugs

  1. Portstack type-to-filter hijacks browser shortcuts — src/main.js:1851
  The capture-phase keydown does e.preventDefault() on any single printable char
  with no modifier guard:
  if (e.key.length === 1 && /[a-z0-9 +]/i.test(e.key)) { e.preventDefault(); ...
  While portstack is open, ⌘R/Ctrl+R (reload), ⌘C (copy), ⌘K, Ctrl+L etc. are
  all swallowed into the filter string. Add if (e.metaKey || e.ctrlKey || 
  e.altKey) return; before this branch.

  2. prefers-reduced-motion is only partially honored — src/main.js:562, 810, 
  916
  The one-shot if (motion.reduced) renderer.render(scene, camera) lines imply
  the WebGL scenes should be static, but their updaters are pushed
  unconditionally and the master loop runs regardless — so reduced-motion users
  still get the full icosa deformation, orbit rig, shader layer, and hero
  parallax. Only the marquee (main.js:611) and pixel arc (main.js:994) actually
  check motion.reduced. Also scrollToId's fallback uses scrollIntoView({ 
  behavior: "smooth" }), which should be auto when reduced.

  3. Snapshot captures a blank fx layer — src/main.js:1279
  ctx.drawImage(fxCanvas, …) reads from a WebGL canvas created without
  preserveDrawingBuffer: true (main.js:904). Outside the frame in which it was
  rendered, the drawing buffer is typically cleared, so the shader layer
  silently comes out empty/transparent in saved PNGs. Either pass
  preserveDrawingBuffer: true to that renderer (it's low-power/small anyway) or
  force a synchronous renderer.render() immediately before drawImage.

  4. Synth toggle leaks updaters — src/main.js:1271
  toggleSynth does updaters.push(...) inside the enable path, so every off→on
  cycle adds another (guarded, but never removed) updater. Register it once at
  module scope.

  5. Wheel-scrubbing can select a filtered-out node — src/main.js:1761
  With a filter active, only "decisive" wheel deltas (>40) route through
  nextVisible; slow trackpad deltas go through scrubTo, which snaps selected to
  the nearest index regardless of matches(i). Enter then ports to a card that's
  visually dimmed out.

  6. Dead tick click handlers — src/index.css:238 vs src/main.js:1436
  .port-tick { pointer-events: none; } means the click listeners added to each
  tick never fire. Harmless today (the #port-track handler computes roughly the
  same index), but it's misleading — remove either the listeners or the
  pointer-events: none.

  7. Char cascade unit mismatch — src/main.js:836–846
  resetChars sets translateY(112%) but the anime tween runs y: [112, 0] — 112px,
  not 112%. Invisible on the big desktop headline (char box ≈ 112px), but
  they're different quantities; use y: ["112%", "0%"] to match the reset.

  Cosmetic / copy inconsistencies

  - Registry footer says "15 NODES ONLINE" (index.html:254) but the registry
  renders 17 rows (20 modules − top/footer/outro).
  - Preloader says "MODULE 01/16" (index.html:113), footer says "16 LIVE"
  (index.html:1156), HUD shows the real 20.
  - Hotkeys overlay lists NEXT/PREV MODULE twice (index.html:52–53 and 58), and
  "MODULE 10 → 0" actually jumps to IMPULSE which is numbered 09.
  - Snapshot watermark hardcodes #c8ff2e (main.js:1280) while everything else is
  theme-aware.
  - Glyph storm's exclusion list (main.js:1311) omits #portstack, so it
  scrambles the overlay's card text while the transport is open.

  Performance notes

  - Pixel arc (main.js:951–998) runs its own always-on rAF doing up to ~25k
  fillRect calls/frame with per-pixel globalAlpha, even when the page is idle
  and nothing scrolls. It's the single heaviest thing here — consider dropping
  to half rate when motion.velocity ≈ 0 and body.idle, or pre-rendering rows to
  an offscreen canvas.
  - Glyph storm walks every text node in <body> and calls getBoundingClientRect
  + getComputedStyle per candidate every ~600ms (main.js:1307–1317) — a
  guaranteed forced layout. Caching the pool and refreshing it on scroll-end
  would cut most of it.
  - Per-element window-level pointermove listeners in magnetic/spotlight each
  call getBoundingClientRect on every move (main.js:741, 755); fine at this
  scale, but worth knowing it's O(cards) per event.
  - The single-file build inlines four font families (Space Grotesk, JetBrains
  Mono, Plaster, Bitcount — the latter two used only in the preloader lockup) as
  base64. Expect a multi-MB HTML; subsetting or dropping the two decorative
  fonts would help a lot.

  Cleanup

  - Unused dependencies: react, react-dom, @react-three/fiber, framer-motion,
  lucide-react, clsx, tailwind-merge are never imported (the app is vanilla JS),
  and the react() plugin in vite.config.ts:13 is unneeded. They won't bloat the
  bundle (never imported) but they bloat installs and imply a stack that isn't
  there.
  - Dead code: scrollToModuleByIndex (main.js:392) is never called; spring is
  imported but unused (main.js:4); let x = reverse ? 0 : 0 (main.js:606); the
  body.booted .w-in / .h-fade CSS transitions (index.css:93–96) are superseded
  by the anime-driven boot entrance and could go.
  - layoutCards declares const dim = … (main.js:1467) shadowing the #port-dim
  element variable — works, but rename one.

  What's notably good

  - The busy transition guard + capture-phase keyboard scoping in portstack is
  the right pattern, and stopPropagation in capture correctly starves the
  page-level hotkey handler.
  - Fractional deckPos easing with time-based Math.min(1, easeK * dt) is
  frame-rate independent — correctly done.
  - No XSS surface: the only user-typed strings (cmdk query, port filter) go
  through textContent/predicate matching, never innerHTML.
  - Drag module physics (pointer-capture, velocity sampling with dt floor, wall
  restitution) is clean and correct.

  The four things I'd actually fix before shipping: the modifier-key guard (#1),
  reduced-motion gating (#2), the snapshot buffer issue (#3), and the
  unused-deps purge.

● Pulled the site apart (it's a Next.js build, but the SSR HTML + inline styles
  reveal most of its tricks). Funny coincidence: their accent is #C0FE04 —
  nearly identical to our #c8ff2e acid. The aesthetic kinship is strong (mono
  captions, uppercase micro-labels, HUD-frame layout), so these inspirations
  transfer naturally.

  What defines haoqi.design

  - Two WebGL canvases: one fixed behind content (background scene) and one
  fixed on top of everything (z-30, pointer-events-none) — a cursor-reactive
  overlay layer above the DOM
  - HUD as a picture frame: logo top-left, nav top-right (WORK / CONTACT / 
  THEME[A] / SOUND[|]), bottom edge has local time GMT+8 CN --:-- and a live
  cursor coordinate readout 0001 X 0001 Y
  - Dotted-border hover language — every interactive chip gets a 2px dotted
  outline on hover, like a Figma annotation
  - Hand-drawn SVG signature that draws itself in via stroke-dashoffset (acid
  green, staggered per-path delays, reduced-motion fallback included)
  - Passcode-protected redacted text — ■■■■■■ inline in a sentence; click →
  enter passcode → reveals the company name
  - Pinned sticky statement ("INNOVATE WITH PURPOSE") in a scroll-driven section
  - Variable-font width axis (wdth: 120) for the condensed display type
  - Full light/dark/system theme cycle, ambient sound toggle in the header,
  fullscreen page transitions

  What I'd add to ours, ranked

  1. Port wipe transition (their FullscreenTransitionProvider) — highest impact.
  Right now portstack/palette jumps scroll you across the page with Lenis — you
  see the travel. Instead: shutter closes (two acid-lined panels or a scanline
  sweep), scroll immediate: true while hidden, shutter opens at the destination.
  Makes every jump feel like a teleport and hides the weakest part of
  long-distance navigation. Hooks straight into confirmPort() and jumpTo().

  2. Redacted / classified block — best thematic fit.
  We already have scramble(). Add a ▓▓▓▓▓▓▓ redacted line in the outro or
  registry ("MODULE +? — CLASSIFIED"), click it → inline passcode prompt (or
  unlock via Konami since we have overdrive) → glyph-scramble decode into the
  hidden content. Could even gate a secret 21st module. This is free given our
  glyph infrastructure and it's the kind of toy people screenshot.

  3. Cursor coordinate readout in the HUD.
  X 0431 · Y 0212 in the bottom telemetry bar, updating from motion.px/py we
  already track. Ten lines of code, pure aesthetic profit — and it makes the
  page feel instrumented even when idle. Bonus: clicking it could toggle our
  debug grid.

  4. Self-drawing SVG scribble.
  anime.js v4 ships createDrawable / svg module (I verified it's in the 4.5.0
  exports). Add a hand-drawn underline/circle scribble on the hero headline or
  an "approved" signature stamp in the outro that draws on when scrolled into
  view. Theirs is one of the site's most memorable moments and costs ~40 lines.

  5. Velocity → variable font weight.
  They use the wdth axis statically; we can go further because our whole concept
  is motion-driven. Space Grotesk Variable has a wght 300–700 axis — map
  |motion.velocity| to font-variation-settings: "wght" on the hero or LERP
  headline. Type that physically fattens when you scroll fast is a perfect
  StringTune demo module ("11 / STRINGWEIGHT").

  6. Pinned typographic statement section.
  A position: sticky full-viewport section where "THE WEB / NOW IN / MOTION"
  reveals word-by-word (or fills via background-clip: text) as you scroll
  through a 300vh wrapper. We have sticky machinery in PROGRESS/SEQUENCE but no
  pure-typography pinned moment — it would pace the page nicely between the
  demo-dense sections.

  7. Dotted-annotation hover language.
  Adopt their dotted-outline hover on our HUD chips (glyph toggle, PORT button,
  theme swatches, footer buttons). One CSS utility class, unifies the
  "engineering annotation" feel we're already going for.

  8. Surface the synth + local time.
  Their SOUND[|] sits in the header; our synth hides behind M. Add a HUD toggle
  with a tiny animated level indicator. And next to the UTC clock, show
  visitor-local time (GMT+X) — we're a telemetry HUD, we should know where the
  operator is.

  9. Full light theme (THEME[A] cycle).
  Our 6 themes only swap the accent. A real inverted "paper" mode (bone
  background, ink text) cycled via T or a dedicated [A] control is the biggest
  visual differentiator on this list — but also the most work, since a lot of
  our colors are hardcoded in Tailwind classes, canvas fills (#030308), and
  shader uniforms. Worth doing only if you want the flex.

  10. Foreground WebGL overlay layer.
  Their top-layer canvas puts effects above the DOM. We render everything behind
  (#fx-layer is z-0). A subtle screen-blended pointer trail (ink smear / heat
  ripple following motion.spx/spy) on a z-[75] canvas would add depth our stack
  currently lacks — and we already have the renderer patterns to do it cheaply.

  My picks if you want to start now: 1, 2, 3, 4 — the wipe transition changes
  how the whole site feels, and the other three are high-charm/low-cost. Want me
  to implement any of them?

● Build passes. Here's what changed — the abstract bars/wave/arc placeholders

  Live previews (makeLivePreview)

  On summon, each card's preview area gets a deep clone of the real section DOM
  (section.cloneNode(true)), snapshotted before the sections collapse:

  - Clones render into a 1280px-wide virtual page, then sizeLivePreviews()
  measures each card and scales the clone down with transform: scale() to fit
  exactly — so a card shows the module the way it actually looks at desktop
  width, cropped to card height
  - All ids and data-module attributes are stripped from clones so they can't
  collide with the live document, the IntersectionObservers, or the glyph storm
  (which I also explicitly excluded from #portstack)
  - WebGL/2D canvases clone blank, so they're swapped for a wire-grid
  placeholder that tints acid on the front card
  - Scroll-reveal states are force-woken in the clone (.rv → .in, chars
  un-hidden, inline opacity cleared) so previews don't show half-revealed
  sections
  - A CRT scanline overlay sits on every preview permanently — keeps the "live
  monitor feed" read

  Fake loading

  Each preview boots behind a veil: sweeping acid scan-line, SYNC +3D label, a
  000% counter and progress bar. Counters resolve staggered outward from the 
  selected card (240ms + 70ms/step + jitter, each taking ~420–740ms), so the
  deck appears to stream in around your position — near cards lock in first,
  distant ones finish last. When a counter hits 100% the veil fades out and the
  page is revealed. Reduced motion skips straight to ready.

  Re-summoning the deck rebuilds and replays the loading sequence (it already
  rebuilt each time, so the fake boot happens on every open — which I think is
  what you want for the effect). Previews also re-fit on window resize while
  open.

  One honest caveat: these are DOM snapshots, not live-updating feeds — the
  marquee/dials inside previews are frozen at summon time. Animating 20 scaled
  page clones per frame would tank FPS; the scanlines + staggered sync do the
  "live" storytelling instead. If you want, the front card only could get a slow
  refresh (re-clone every second) — say the word.
